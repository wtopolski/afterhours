Koog AI Agents

Kotlin samples exploring the Koog AI agents framework (v1.0.0) with local LLMs via LM Studio and Ollama — interactive chat, custom tools, structured output, and a live bike-sharing API integration.

Overview

What this is

A small set of self-contained Kotlin samples built on top of Koog's node-graph agent runtime. Each file explores one concept — from a plain chat loop to annotation-based tool sets, JSON-schema-validated structured output, and a live Ktor HTTP tool.

Architecture

Node-graph strategy pattern

Koog agents are directed graphs of typed nodes. Edges pick the next node based on the LLM's response — a text message finishes the run, tool calls route through an execute/parse chain.

// Strategy flow
nodeStart → nodeLLMRequest → (text)       → nodeFinish
                            → (tool calls) → nodeExecuteTools → nodeParseResult → nodeFinish
  • Strategy — a directed graph of typed nodes connected by conditional edges.
  • Nodes — typed Input → Output units. Built-in helpers: nodeLLMRequest, nodeExecuteTools.
  • Edges — onTextMessage { } / onToolCalls { } conditions select which path to take.
  • Tools — implement Tool<Args, Result> with a ToolDescriptor, or use @Tool annotations on a ToolSet.
  • Structured output — requestLLMStructured<T>() inside llm.writeSession { } asks the LLM to emit a JSON-schema-validated T.

Known Issues

Quirks worth knowing

Qwen reasoning content

LM Studio routes Qwen's thinking output into reasoning_content and leaves content empty. This breaks two things:

  • Structured output — Koog's parser calls .single() on MessagePart.Text and fails when there are no text parts. Fix: promote reasoning_content to a Text part when no text is present.
  • Tool calls — if reasoning is promoted unconditionally, messages with both reasoning and tool calls end up with a Text part, causing onTextMessage to fire before onToolCalls. Fix: skip promotion when MessagePart.Tool.Call parts are already present.

Both fixes live in LmStudioOpenAIClient.promoteReasoningToText() in utils.kt.

Direct parsing vs. requestLLMStructured

requestLLMStructured<T> is suited for having the LLM generate structured data. When a tool already returns typed objects (e.g. BikeResult), pass the data through directly by mapping ReceivedToolResults in a custom node — letting the LLM reformat it risks truncation.

Setup

Getting started

LM Studio

Primary backend. Install from lmstudio.ai, load a model, start the local API server (default http://localhost:1234).

model: qwen/qwen3.5-35b-a3b

Ollama

Alternative backend. Pull a model, then switch ollama_model in utils.kt.

ollama pull gpt-oss:20b
# Build & test
./gradlew build
./gradlew test

Run samples from your IDE by executing each file's main(), or via the application plugin with the fully-qualified class name:
